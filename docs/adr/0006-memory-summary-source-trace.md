# ADR-0006: Memory 摘要与来源追踪

**状态**: 已接受  
**日期**: 2026-06-19

## 背景

长对话中，历史消息众多：

- 短期记忆需要摘要以节省 Token
- 跨会话查询需要回溯原始消息
- 删除消息后相关摘要需要失效
- Memory 来源必须可追踪

## 决策

采用两层 Memory：

**1. 短期 Memory (ConversationWorkspace)**
- 最近 N 条消息 + 摘要
- 8～12 条或 Token 阈值触发摘要
- MemorySegment (原消息 ID 列表 + 摘要)
- 删除原消息后标记摘要为 stale

**2. 长期 Memory (跨会话)**
- ConversationSummary: 按会话摘要
- MemoryPreference: 用户偏好
- 两级检索：先摘要后原文

**来源追踪**:
- `source_message_ids`: 摘要涵盖的原消息 ID
- `source_trace`: 生成摘要时的 Trace ID
- 删除消息时标记相关摘要为 "needs_rebuild"

## 替代方案

1. **完整历史**: 缺点是 Token 消耗高。
2. **只保留摘要**: 缺点是无法回溯细节。
3. **向量化后删除原文**: 缺点是安全问题。

## 后果

### 优点

- ✅ Token 效率高
- ✅ 可回溯原始信息
- ✅ 支持增量更新
- ✅ 删除策略清晰

### 缺点

- ⚠️ 需要额外的生成调用
- ⚠️ 需要处理摘要失效和重建
- ⚠️ 来源追踪增加存储开销

## 不变量

1. 摘要必须包含来源消息 ID
2. 删除消息后相关摘要需要标记
3. 摘要的摘要不允许（最多两层）
4. 来源消息被删除后，摘要不再自动更新

## 验收标准

- [ ] 摘要事实保持率 ≥ 90%
- [ ] 原消息追溯 100%
- [ ] 删除消息后摘要标记失效
- [ ] 跨会话检索 Recall@5 ≥ 85%
