# 统一对话式论文 Agent：自顶而下执行计划

> 文档状态：开发实施基线 V2  
> 开发方式：先完成整体代码框架和模块契约，再逐模块补充真实实现  
> 关联文档：[01-技术栈文档.md](./01-技术栈文档.md)、[02-产品架构文档.md](./02-产品架构文档.md)

---

## 1. 执行策略

本项目不采用“先做一个功能很少的 MVP，再不断向原代码中追加模块”的方式。

采用自顶而下开发：

```text
产品与系统架构
 → 模块边界
 → 数据模型和接口契约
 → 所有模块的代码骨架
 → Fake/Mock 端到端控制流
 → 逐模块替换为真实实现
 → 全系统集成、评测与部署
 → 独立开展模型 SFT/RL
```

核心原则：

1. 在实现具体能力前，先建立完整目录、接口、状态和依赖关系。
2. 所有模块先提供可调用的 Stub/Fake，再补真实实现。
3. 上层只能依赖接口，不能依赖某个具体模型、数据库或解析器。
4. 每完成一个真实模块，都重新运行全系统契约测试和端到端回归。
5. 系统开发与模型训练相互独立。
6. 不因某个模块尚未完成而临时绕过既定架构。
7. 所有文件、任务、Memory 和模型调用都必须可追踪。

# 4. P0：架构基线

## 4.1 目标

在写业务代码前，确定整个系统的边界和不可随意改变的概念。

## 4.2 需要准备

- 技术栈文档。
- 产品架构文档。
- 核心用户场景。
- 目标硬件和部署环境。
- 初始论文测试材料。
- 数据隐私和版权边界。

## 4.3 具体任务

1. 统一术语：
   - Conversation
   - Message
   - Task
   - Plan
   - Step
   - Skill
   - Tool
   - Sub-agent
   - Memory
   - Workspace
   - Artifact
   - Model Profile

2. 绘制模块依赖图。
3. 定义 Agent 状态机。
4. 定义主要用户流程。
5. 编写架构决策记录 ADR：
   - 为什么使用自定义状态机。
   - 为什么使用 PostgreSQL + pgvector。
   - 为什么使用 SSE。
   - 为什么使用 Model Profile。
   - 为什么会话工作空间和执行沙箱分离。
6. 定义非功能目标：
   - 安全。
   - 可恢复性。
   - 可追踪性。
   - 性能。
   - Memory 质量。
   - 引用准确性。

- 本阶段不实现功能。
- 模块边界应以职责划分，不能按页面或具体模型划分。
- 不在架构中硬编码 Qwen、MinIO 或某个解析器。
- 区分 Workspace、Memory 和沙箱。

## 4.5 验收标准

- 三份项目文档概念一致。
- 所有核心模块均有明确职责和输入输出。
- 不存在两个模块共同拥有同一数据真值。
- 每项外部基础设施都有抽象 Port。
- 系统与模型训练可以独立开发。

## 6.3 Fake 实现要求

Fake 不能只是返回 `None`，而应返回可用于贯通流程的确定性数据。

例如：

- FakeLLM 根据输入标记返回固定 AgentDecision。
- FakeRetriever 返回预置证据。
- FakeTool 返回固定 ToolResult。
- FakeMemory 返回相关历史片段。
- FakeDocumentParser 返回固定结构化论文。
- FakeSubAgent 返回固定 Paper Card。

## 6.4 验收标准

- 所有模块均可被导入。
- API 和 Worker 可以启动。
- 依赖注入可以在 Fake/Real 间切换。
- 项目不存在循环依赖。
- 所有公开方法都有类型和文档。
- 所有模块至少有一条骨架测试。

# 7. P3：Fake 端到端控制流

## 7.1 目标

在任何真实 PDF 解析、模型推理和向量检索尚未实现前，先证明整个系统控制流正确。

## 7.2 必须贯通的场景

### 场景 A：信息充分

```text
用户消息
 → 创建 Task/TaskWorkspace
 → RequirementCheck 通过
 → Skill 选择
 → 计划
 → Fake Tool
 → Verification
 → 最终回答
```

### 场景 B：需要澄清

```text
用户请求不完整
 → Clarifier 提出问题
 → Task 进入 WAITING_USER
 → 用户回答
 → Requirement Brief
 → 原 Task 恢复
 → 后续执行
```

### 场景 C：子 Agent

```text
主 Agent
 → 创建多个 Fake 子 Agent
 → 并发结果
 → 汇总
```

### 场景 D：失败和恢复

- Tool 超时。
- 子 Agent 部分失败。
- 核验失败后重规划。
- 用户取消。
- Worker 重启后恢复。

### 场景 E：Memory 与 Workspace

- 后续对话找到之前消息。
- 后续对话找到之前 Task 生成的文件或脚本。
- 删除后不再检索。

## 7.3 验收标准

- 所有控制流均能通过 Fake 运行。
- 前端可展示所有任务状态。
- Task 可以暂停、恢复、取消和重试。
- 每一步均产生 Trace 和事件。
- Fake 模块可逐个替换，不影响其他模块。

本阶段完成后，才开始真实模块实现。

# 19. 每次模块替换的操作流程

当把 Fake 模块替换为真实模块时：

1. 锁定模块 Port，不临时改调用方。
2. 使用固定 Fixture 完成真实实现。
3. 运行该模块契约测试。
4. 运行直接上下游集成测试。
5. 在开发环境通过配置切换到真实实现。
6. 运行所有 Fake 端到端场景。
7. 运行相关质量评测。
8. 对比性能和错误。
9. 更新 ADR、配置和文档。
10. 将真实实现设为默认，保留 Fake 用于测试。

如果真实实现迫使多个上层模块同时修改，应先检查 Port 是否设计错误，而不是直接传播改动。
