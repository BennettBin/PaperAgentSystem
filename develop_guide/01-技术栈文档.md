## 阶段 C 基础设施实现基线（2026-06-20）

- PostgreSQL Adapter 使用 SQLAlchemy 2 同步 Session、Repository Port 和 Unit of Work；Alembic 空库迁移创建 pgvector 扩展及核心实体、队列、对象和任务事件表。
- 所有 Workspace 归属资源的 Repository 查询必须包含 `workspace_id`，软删除记录默认不可见；版本字段使用 SQLAlchemy 乐观锁。
- Redis Adapter 提供四类任务队列、取消标记、分布式锁、重试/死信、崩溃恢复和临时事件通知。任务历史与最终状态保存在数据库，Redis 不是状态真值。
- Celery 配置启用 late acknowledgement、worker lost reject、每 worker 单条预取和独立队列路由。
- MinIO Adapter 使用 `uploads`、`workspace`、`artifacts` 三类逻辑 Bucket；对象按 Workspace 和 SHA-256 组织，数据库维护上传完成状态与引用计数。
- 文件上传同时检查 MIME 与签名，写入采用临时对象后复制为正式对象；失败时清理临时对象和数据库记录。
- SSE 使用数据库 `task_events` 作为真值、Redis 作为唤醒通知；客户端以 sequence/`Last-Event-ID` 续传，终态事件发送后关闭连接。

## 阶段 D 会话、Workspace 与 Memory 实现基线（2026-06-20）

- Conversation/Message 应用服务通过 Repository 完成搜索、稳定排序分页、附件关联、单条/范围删除和级联失效；所有操作强制 Workspace 隔离。
- ConversationWorkspace 创建 `uploads/shared/tasks/artifacts`；TaskWorkspace 创建 `inputs/scratch/scripts/outputs/logs`。本地目录仅为执行视图，Manifest、数据库 WorkspaceEntry 和 MinIO 对象键用于恢复。
- Workspace 路径仅接受受控 POSIX 相对路径，拒绝绝对路径、`..`、反斜杠和符号链接逃逸；脚本写入后默认无执行位。
- Workspace 搜索保存文件名、提取文本、摘要、Embedding 和来源字段，并按当前 Task、当前 Conversation、历史 Conversation 分层排序。
- 短期 Memory 使用最近消息窗口和 MemorySegment；检索先定位摘要，再按 source_message_ids 回读原始消息。消息删除会失效相关 Segment。
- 长期 Memory 使用 ConversationSummary 和显式保存的 MemoryPreference；跨会话摘要与历史 WorkspaceEntry 联合检索，删除/遗忘同步清除摘要和索引。


## 1. 技术目标

本项目要实现一个类似 ChatGPT/Codex 交互方式的论文 Agent：

- 用户只面对一个对话窗口。
- 用户可在对话中上传 PDF、DOCX、Markdown、LaTeX 等文件。
- 用户可从会话列表查看历史聊天及其附件，并按需删除消息或整个会话。
- 系统自动识别目标、选择 Skill、制定计划、调用 Tool。
- 复杂任务可以拆成多个子任务，并由子 Agent 并行处理。
- 系统具有短期与长期记忆，并能从摘要追溯到原始消息和历史文件。
- 所有事实性论文回答尽可能附带原文证据和页码。
- 后台执行过程可取消、可恢复、可追踪、可评测。
- 主要生成模型限定为 1.5B/1.7B 和 4B 开源小模型。
- 支持本地部署、4-bit 量化和后续 QLoRA/SFT。

技术选择遵循四个原则：

1. **可控优先**：核心 Agent Runtime 使用显式状态机和严格 Schema。
2. **可替换优先**：模型、向量模型、解析器和存储均通过接口隔离。
3. **先闭环再扩展**：先完成单机 Docker Compose，再考虑分布式部署。
4. **评测驱动**：每项组件替换都必须在固定测试集上比较。

## 3. 前端技术栈

### 3.1 Next.js + TypeScript

选择原因：

- 适合构建单页式对话应用。
- React 生态中已有成熟的 Markdown、代码块、PDF 和流式 UI 组件。
- TypeScript 能约束前后端事件、任务状态和消息结构。
- 支持用户登录、会话列表、历史文件预览和 Memory 设置页面。

前端不承担 Agent 推理，只负责：

- 会话与消息展示。
- 历史会话搜索、分页加载和按需删除。
- 附件上传和文件状态。
- 查看当前会话及历史会话关联的文件。
- 发送用户请求。
- 消费 SSE 事件流。
- 显示任务步骤的公开状态。
- 取消、重试和继续任务。
- 展示引用、表格和生成文件。

### 3.2 状态管理

MVP 推荐：

- 服务端状态：TanStack Query。
- 局部 UI 状态：React State 或 Zustand。
- 不使用大型 Redux 架构。

状态分类：

| 状态 | 存放位置 |
|---|---|
| 会话、消息、文件、任务 | TanStack Query 缓存 |
| 历史会话搜索、Memory 检索结果 | TanStack Query 缓存 |
| 输入框、弹窗、展开项 | React/Zustand |
| Agent 执行真值 | 后端数据库，不由前端维护 |

### 3.3 流式通信

首选 SSE，而不是 WebSocket：

- 主要数据方向是服务端向浏览器推送任务事件。
- 实现、重连和代理配置更简单。
- 用户发送、取消、确认仍使用普通 HTTP API。

事件示例：

```json
{
  "event_id": "evt_001",
  "task_id": "task_001",
  "sequence": 12,
  "type": "tool_completed",
  "title": "论文解析完成",
  "data": {
    "tool_name": "parse_document",
    "file_id": "file_001"
  },
  "created_at": "2026-06-18T20:00:00Z"
}
```

SSE 客户端必须支持：

- 使用 `Last-Event-ID` 断线续传。
- 对 `sequence` 去重。
- 任务结束后关闭连接。
- 页面刷新后从数据库恢复任务状态。

### 3.4 UI 组件

建议组件：

- ConversationSidebar
- MessageList
- MessageComposer
- AttachmentPicker
- TaskProgress

- ArtifactCard
- FilePreview
- ConfirmationDialog

用户可以看到“正在检索”“正在核验”等公开状态，但不显示模型隐藏推理文本。

## 4. 后端 API 技术栈

### 4.1 FastAPI

选择原因：

- Python 与模型、RAG、PDF 解析和训练生态一致。
- Pydantic 适合 Tool 参数、Agent 决策和 API 的 Schema 验证。
- 原生支持异步 API、文件上传、SSE/流式响应和 OpenAPI。
- 便于将模型调用、任务队列和数据库封装成服务。

### 4.2 Python 环境

建议：

- Python 3.12。
- 使用 `uv` 管理依赖与虚拟环境。
- 使用 Ruff 进行格式化和静态检查。
- 使用 mypy 或 Pyright 做类型检查。

### 4.3 后端代码分层

```text
api/
├── routes/          # HTTP/SSE 路由
├── schemas/         # API Pydantic Schema
├── services/        # 会话、文件、任务与 Memory 应用服务
├── repositories/    # 数据库访问
├── auth/            # 身份与权限
└── dependencies/    # FastAPI 依赖注入
```

规则：

- Route 不直接执行 Agent。
- Route 创建任务并立即返回 `task_id`。
- Agent 与解析任务由 Worker 执行。
- Route 不直接拼接数据库 SQL。
- API Schema 与数据库 Model 分离。

## 5. Agent Runtime 技术方案

### 5.1 MVP 不直接依赖通用 Agent 框架

首版采用自定义、可持久化的有限状态机：

```text
RECEIVED
 → UNDERSTANDING
 → REQUIREMENT_CHECK
 ├→ CLARIFYING
 │    → WAITING_USER
 │    → REQUIREMENT_CHECK
 → SKILL_SELECTED
 → PLANNED
 → EXECUTING
 → VERIFYING
 ├→ REPLANNING
 ├→ FAILED
 ├→ CANCELLED
 └→ COMPLETED
```

理由：

- 你需要真正理解 Agent 的运行机制。
- 小模型需要更强的步骤、工具和预算限制。
- 自定义状态便于保存、恢复和评测。
- 将来可在保持接口不变的情况下引入 LangGraph。

### 5.2 Agent Runtime 模块

```text
agent_runtime/
├── orchestrator.py
├── state.py
├── context_builder.py
├── requirement_clarifier.py
├── memory_retriever.py
├── skill_selector.py
├── planner.py
├── executor.py
├── verifier.py
├── replanner.py
├── termination.py
└── policies.py
```

截至 2026-06-20，阶段 E 已实现可测试的 Runtime 核心：检索优先的需求澄清、manifest
粗筛与正文延迟加载、结构化 DAG Planner/Replanner、预算化 Executor、来源可追踪的
ContextBuilder，以及不依赖生成模型的规则 Verifier。Executor 对步骤、Token、总时间、
Tool/子 Agent 调用、取消、超时与幂等结果执行硬限制；完整 Tool 权限和真实子 Agent
生命周期继续由阶段 F 的专用 Runtime 提供。

### 5.3 结构化决策

模型不返回任意自然语言行动，而返回严格 Schema：

```json
{
  "decision": "call_tool",
  "reason_summary": "需要读取上传论文的结构",
  "tool_call": {
    "name": "get_document_outline",
    "arguments": {
      "file_id": "file_001"
    }
  }
}
```

允许的 `decision`：

- `call_tool`
- `spawn_subagents`
- `complete_step`
- `replan`
- `ask_user`
- `final_answer`
- `fail`

### 5.4 需求完整性判断与反问

在选择 Skill 和生成计划前，Requirement Clarifier 判断当前请求是否具备可执行信息。

检查维度：

- 用户最终想获得什么输出。
- 处理对象和文件是否明确。
- 任务范围、章节类型或比较维度是否明确。
- 目标语言、长度、格式、风格和截止条件是否会影响结果。
- 事实材料、实验数据、引用或不可变项是否缺失。
- 是否能从当前会话、Memory、文件或 Tool 中自动获得缺失信息。

只有缺失信息会实质改变执行结果，并且系统无法自行发现时才反问。

反问一次提出 1～5 个相关问题：

```json
{
  "decision": "ask_user",
  "reason_summary": "缺少章节类型、目标篇幅和可用实验结果",
  "questions": [
    {
      "question_id": "q1",
      "question": "你希望撰写论文的哪个章节？",
      "required": true
    },
    {
      "question_id": "q2",
      "question": "目标语言和大致篇幅是多少？",
      "required": false
    },
    {
      "question_id": "q3",
      "question": "哪些实验结果可以作为本章节的事实依据？",
      "required": true
    }
  ],
  "allow_best_effort": true
}
```

规则：

- 优先一次批量提出问题，避免逐个来回询问。
- 不询问能从文件、Memory 或 Tool 获得的信息。
- 问题按重要性排序，最多 5 个。
- 用户可以回答部分问题，也可以选择“按现有信息继续”。
- Task 进入 `WAITING_USER`，保存 Clarification Round。
- 用户回复后，将问题、回答和原需求合并成 Requirement Brief，再次执行完整性判断。
- 默认最多两轮澄清；仍不完整时提供最佳努力结果或说明无法继续。
- 澄清消息属于当前会话 Memory，并与原 Task 关联。

Requirement Brief 保存：

- original_request
- clarified_goal
- target_artifact
- input_files
- constraints
- user_answers
- assumptions
- missing_information
- clarification_round

### 5.5 状态持久化

每执行一个动作都：

1. 读取最新任务状态。
2. 检查任务是否已取消。
3. 记录 Agent 决策。
4. 执行动作。
5. 记录 Observation。
6. 更新计划和预算。
7. 提交事务。

进程崩溃后，Worker 可从最近一个稳定状态继续。

## 7. Tool 技术方案

### 阶段 F 实现基线（2026-06-20）

- `tool_runtime` 使用 Pydantic 2 输入/输出模型；模型参数不能提供
  `workspace_id`、`user_id`、`conversation_id`、`task_id`、权限、确认或 Trace 字段。
- Tool 调用同时受 Registry、当前 Skill 白名单和系统权限约束；高风险 Tool 的确认状态
  只能由系统上下文注入。
- Runtime 统一提供超时、有限重试、任务级幂等键、内联输出上限、`data_ref` 和 Trace。
- Workspace Tool 只接受 `workspace_entry_id` 或受控相对路径，六项能力为
  list/search/read/write/promote/save_artifact；Promote 写审计，持久结果写对象存储。
- 11 个能力型 Skill 在加载时校验 Manifest、SKILL.md、输出 Schema、示例、Tool 白名单、
  逻辑 Model Profile、追问条件、终止条件和验收规则，激活 Trace 记录 Skill 版本。

### 7.1 Tool 基类

```python
class Tool:
    name: str
    description: str
    input_model: type[BaseModel]
    output_model: type[BaseModel]
    permission: str
    timeout_seconds: int
    retryable: bool

    async def execute(self, context, arguments):
        ...
```

### 7.2 Tool 执行规则

- 模型只能调用 Tool Registry 中注册的 Tool。
- Skill 进一步缩小本轮允许的 Tool 白名单。
- 所有参数经过 Pydantic 验证。
- Tool 只能使用内部 `file_id`，不能接受任意系统路径。
- Tool 通过 `workspace_entry_id` 访问任务工作空间，不能自行拼接路径。
- Tool 返回大小受限；大结果写入 Artifact 或对象存储。
- 只读 Tool 可自动执行。
- 有外部副作用的 Tool 必须经过确认策略。

## 8. 后台任务和子 Agent

### 8.1 Celery + Redis

Celery 用于：

- 论文解析。
- OCR。
- Embedding。
- 主 Agent 任务。
- 子 Agent 并行任务。
- 批量离线评测。

Redis 用于：

- Celery Broker。
- 短期缓存。
- 任务取消标记。
- 分布式锁。
- SSE 临时事件通知。

数据库仍然是任务状态真值；Redis 丢失不能导致任务记录消失。

### 8.2 子 Agent 实现

子 Agent 本质上是具有以下约束的子任务：

- 独立 `task_id` 和 `parent_task_id`。
- 独立上下文。
- 一个明确目标。
- 一个 Skill。
- 一个逻辑 `model_profile`，不保存物理模型路径。
- Tool 白名单。
- 最大步骤和超时。
- 不能直接向用户发消息。

第一版只实现一种子 Agent：`paper_reader_agent`。开发阶段可让它使用基础 4B；训练完成后只修改 Model Registry，将其切换到论文阅读专用 SFT/RL Adapter。

阶段 F 的 `paper_reader_agent` 只读取一个系统分配的 `file_id`，使用独立
`paper_reader_v1` Profile、最多 8 步预算和固定 Tool 白名单，返回 Paper Card、证据列表
和缺失字段，不直接产生用户消息。Sub-agent Manager 将每个子任务以
`parent_task_id`/`child_task_id` 持久化，生产调度使用 Celery Group，并限制并发和最大
嵌套深度 1；批次允许部分失败，主任务取消会撤销 Group 并取消未启动子任务。

### 8.3 后台执行位置

Agent 主任务、子 Agent、解析和文件 Tool 运行在独立 Worker 容器中：

```text
FastAPI 创建 Task
 → Celery 分配 Worker
 → Worker 创建/挂载 TaskWorkspace
 → Agent Runtime 与 Tool 执行
 → 中间内容写入任务目录
 → 需要持久化的条目同步到 MinIO
 → PostgreSQL 更新 Manifest 和状态
```

API 进程不执行长任务，模型服务也不直接访问工作空间。模型只能通过 Tool 返回的受控内容了解文件。

## 9. 数据库与存储

### 阶段 G 论文解析与 RAG 实现基线（2026-06-20）

- PDF 使用 PyMuPDF 提取逐行文本、1-based 页码、bbox、字体大小和阅读顺序；重复边缘文本
  识别为页眉页脚，章节结合字号与标题规则识别，并输出可解释质量评分。
- 文本不足的页面触发 OCR；PaddleOCR 和 Tesseract 均通过可替换 Adapter 接入，记录
  页面、引擎、置信度、回退和低质量警告。OCR 失败不会静默写入空索引。
- Chunk 使用 Section 父块和受限长度子块，保存 section path、page range、bbox、
  source block IDs、parent、previous 和 next。文件 checksum 保证幂等 Embedding，
  删除文件时同步删除 ParsedDocument 和 Chunk。
- PostgreSQL 使用 `vector(1024)`、HNSW cosine 索引和 `to_tsvector('simple', ...)` GIN
  索引；混合检索执行查询改写、Vector Top-30、Keyword Top-30、RRF 和 Reranker Top-8，
  所有查询强制 Workspace，并可进一步限制 file_id。
- 引用回答只从检索证据句生成 Claim，Citation ID 由程序按 `C1...Cn` 分配，携带
  file_id、页面和 bbox。Claim-Evidence 检查失败或查询覆盖不足时明确拒答。

### 9.1 PostgreSQL

保存：

- 用户和工作区。
- 会话和消息。
- 消息区间摘要、会话级摘要和摘要到原消息的映射。
- 用户明确保存的长期偏好。
- 文件元数据。
- Agent 任务和计划。
- Tool 调用。
- 子 Agent。
- Trace 与评测。
- 文档结构和文本块元数据。

### 9.2 pgvector

选择原因：

- MVP 不需要额外维护一套向量数据库。
- 向量和文档元数据可以在同一事务系统中管理。
- 查询可强制包含 `workspace_id`、`file_id` 等权限过滤。
- 支持精确和近似向量检索，包括 HNSW/IVFFlat。

MVP 先使用 HNSW；数据量很小时可先精确搜索作为正确性基线。

### 9.3 PostgreSQL 全文检索

MVP 使用 PostgreSQL Full Text Search 构建关键词检索，并与向量结果做 RRF 融合。

当满足以下条件时再引入 OpenSearch：

- 文本块达到百万级。
- 需要复杂中文分词或高级查询语法。
- PostgreSQL 关键词检索已成为明确瓶颈。

### 9.4 对象存储

MinIO 保存：

- 用户原始文件。
- 解析后的 JSON。
- 页面图片。
- OCR 中间结果。
- 导出的 Markdown/DOCX。
- 大型 Trace Artifact。

数据库只保存对象 Key、哈希、MIME、大小和权限信息。

### 9.5 Memory 存储与检索

Memory 采用“原始记录 + 分层摘要 + 向量/关键词索引”，不能把所有历史消息无限拼接进模型上下文。

#### 短期记忆

短期记忆限定在当前会话，包括：

- 最近消息滑动窗口。
- 当前任务目标、计划和中间结果。
- 较早消息的区间摘要。
- 当前会话上传或引用过的文件。

满足任一初始阈值时，由 Celery 异步创建一个消息区间摘要：

- 新增 8～12 条消息。
- 未摘要内容达到约 4K～6K tokens。
- 预计上下文超过模型预算的 60%。

每条摘要保存：

```json
{
  "summary_id": "summary_001",
  "conversation_id": "conv_001",
  "start_message_id": "msg_001",
  "end_message_id": "msg_012",
  "summary_text": "用户上传了三篇论文，并确认重点比较实验设计……",
  "entities": ["论文 A", "数据集 X"],
  "decisions": ["比较方法、数据集和指标"],
  "file_ids": ["file_001", "file_002"],
  "summary_model": "model/version",
  "source_message_count": 12
}
```

当前会话检索流程：

```text
当前问题
 → 保留最近消息
 → 检索当前会话的区间摘要
 → 选择 Top-K 摘要
 → 根据消息 ID 范围回读原始消息
 → 对原始消息重排和裁剪
 → 构建模型上下文
```

摘要只负责定位，原始消息是最终事实来源。

#### 长期记忆

长期记忆跨会话存在，包括：

- 历史会话的会话级摘要。
- 历史会话的区间摘要和原消息映射。
- 用户明确允许保存的偏好、术语和写作约束。
- Workspace 中仍未删除的上传文件及其解析索引。
- 用户保留的 Artifact。

跨会话检索采用两级流程：

```text
当前问题
 → 检索会话级摘要和长期偏好
 → 找到相关 conversation_id
 → 检索该会话的区间摘要
 → 回读对应原始消息
 → 按需读取历史文件或论文索引
```

默认优先当前会话。只有用户提到“之前”“上次”“我上传过的文件”，或当前任务确实需要历史信息时，才扩大到跨会话检索。

#### 历史文件

文件属于 Workspace，通过 `conversation_files` 关联表记录它在哪些会话被上传或使用：

- 用户可在后续会话中查看和重新关联仍保留的历史文件。
- Agent 通过 `search_workspace_files` 按文件名、元数据、会话和语义查找文件。
- 模型只获得 `file_id`，不能直接访问对象存储路径。
- 文件删除后，Document、Chunk、Embedding、摘要引用和缓存必须失效。

#### 摘要任务

摘要 Worker 执行：

1. 锁定尚未摘要的稳定消息区间。
2. 读取原始消息和关联文件元数据。
3. 生成结构化摘要。
4. 检查关键事实、文件引用和消息范围。
5. 写入摘要、全文索引和 Embedding。
6. 更新 `summarized_through_message_id`。

同一区间使用唯一约束，防止重复摘要。

#### 删除与遗忘

支持：

- 删除单条或选定范围的聊天记录。
- 删除整个会话。
- 单独删除历史文件。
- 删除会话时选择是否同时删除该会话独有的文件。

删除后必须：

1. 立即从界面和检索结果隐藏。
2. 取消相关未完成任务。
3. 删除或重建受影响的区间摘要和会话摘要。
4. 删除对应 Memory Embedding 和 Redis 缓存。
5. 硬删除完成后不可再被 Agent 检索。

共享文件不会因删除一个会话而默认删除；只有用户明确选择或文件没有其他引用时才清理对象和索引。

### 9.6 会话与任务工作空间

系统为每个会话建立一个持久化逻辑工作空间，并为每个后台任务建立独立子目录：

```text
workspace/{workspace_id}/conversations/{conversation_id}/
├── uploads/                       # 上传文件的逻辑引用
├── shared/                        # 后续对话可复用的中间材料
├── tasks/{task_id}/
│   ├── inputs/                    # 本任务使用的文件引用
│   ├── scratch/                   # 可清理的临时文件
│   ├── scripts/                   # Agent 生成的脚本，默认不可执行
│   ├── outputs/                   # 任务输出候选
│   ├── logs/                      # 受控执行日志
│   └── manifest.json
└── artifacts/                     # 用户可见、长期保留的正式结果
```

这里的目录是逻辑结构：

- Worker 执行时，将当前 `tasks/{task_id}` 映射到一个受限本地目录。
- MinIO/S3 是持久化真值，避免 Worker 容器重启后文件丢失。
- PostgreSQL 保存 WorkspaceEntry、对象 Key、哈希、来源、状态和权限。
- 任务结束后，`scratch/` 可自动清理；`scripts/`、`outputs/` 根据保留策略同步到对象存储。
- 用户明确保留的文件可从任务目录提升到 `shared/` 或 `artifacts/`。

#### Manifest

每个任务维护 Manifest：

```json
{
  "task_id": "task_001",
  "conversation_id": "conv_001",
  "entries": [
    {
      "entry_id": "entry_001",
      "relative_path": "scripts/extract_tables.py",
      "kind": "script",
      "mime_type": "text/x-python",
      "sha256": "...",
      "created_by": "tool:generate_script",
      "status": "generated",
      "executable": false,
      "retention": "conversation"
    }
  ]
}
```

所有文件操作使用 `entry_id` 和经过校验的相对路径。禁止 `..`、绝对路径、符号链接逃逸和访问其他会话目录。

#### 工作空间索引

可检索内容：

- 文件名、类型、标签和创建时间。
- 文本型中间文件、Markdown、JSON、日志摘要和脚本源码。
- 文件来源 Task、Skill、Tool 和关联消息。
- 生成文件的内容摘要与 Embedding。

二进制文件先提取元数据或文本，再建立索引。后续对话的检索顺序：

```text
当前会话 shared/artifacts
 → 当前会话历史任务 outputs/scripts
 → 当前任务 scratch
 → 用户明确要求时检索历史会话工作空间
```

Agent 从工作空间取回文件时，ContextBuilder 只加入相关片段和 `entry_id`，不把整个目录放入上下文。

#### 脚本安全

- Agent 可以生成、保存、检索和向用户展示脚本。
- MVP 中脚本默认 `executable=false`，不能由普通 Worker 直接运行。
- 普通论文解析和文件转换只能调用预注册 Tool。
- 未来需要运行 Python、Shell、LaTeX 或论文附带代码时，必须提交到一次性执行沙箱。
- 沙箱只挂载当前任务的 `inputs/` 为只读，并提供独立可写 `scratch/`。
- 沙箱限制 CPU、内存、进程数、时间、输出大小和网络。
- 沙箱输出通过安全扫描后才能写回 `outputs/`。

#### 生命周期

| 区域 | 默认生命周期 |
|---|---|
| uploads | 由用户删除或 Workspace 策略决定 |
| shared | 随会话保留，可由用户删除 |
| task inputs | 任务结束后保留引用，不复制原文件 |
| scratch | 任务完成后 24 小时内清理 |
| scripts | 默认随会话保留，可配置自动清理 |
| outputs | 任务完成后保留 30 天或由用户提升 |
| artifacts | 用户删除前长期保留 |

删除会话时，工作空间条目、索引和对象按引用计数级联清理；仍被其他会话引用的上传文件不删除。

## 阶段 H 论文领域应用基线（2026-06-20）

- `academic_tasks` 在 RAG Evidence 与 Agent Runtime 之间提供确定性事实边界：单论文
  Paper Card、并行多论文比较、Writing Brief/Evidence Map、章节草稿、受保护改写和
  Evidence Matrix 优先综述。
- Paper Card 的研究问题、方法、数据集、指标、结果、贡献和局限必须绑定 Evidence；
  缺失字段保持为空并显式报告。
- 多论文比较直接消费 Sub-agent Manager 的并行 Paper Card 批次；部分失败不丢失成功
  结果。比较数字必须出现在对应 Evidence，综合结论必须携带 Evidence ID。
- 初次撰写只能使用 Writing Brief 中 `allowed_as_fact` 的事实；观点和推测不能当作事实。
  七类章节均先生成段落计划，草稿保存来源、缺失项并标记 `review_required=true`。
- 改写前提取命题、数字、公式、Citation 和保护术语，改写后执行不可变项及语义回归。
- 文献综述先构建 Evidence Matrix；事实 Claim 必须有 Evidence ID，推断必须显式标记。
  这些规则服务通过 Tool Runtime 暴露，生成模型只能在规则边界内替换文本实现。
- 当前质量指标是版本化的确定性 fixture 基线，不代表真人盲评或真实 4B 模型质量；后者
  必须在阶段 J 的独立评测报告中验证。

## 11. 模型与推理服务

### 11.1 生成模型

首批基线：

- Qwen3-1.7B：路由、Skill 初选、查询改写和抽取。
- Qwen3-4B：规划、工具调用、证据回答、综合和润色。

这里的模型名称是基座。系统运行时不由 Skill 直接引用 `Qwen3-4B` 或某个 Adapter 路径，而是引用逻辑 Model Profile。

模型通过统一客户端调用：

```python
class LLMClient:
    async def generate(self, messages, response_schema=None, **options):
        ...
```

业务代码不依赖具体模型名称。

### 11.2 推理服务

GPU 环境首选 vLLM：

- 可提供 OpenAI 兼容 API。
- 适合并发和服务化。
- 便于主模型、路由模型和训练后 Adapter 统一接入。

CPU 或轻量本地环境使用 llama.cpp：

- 运行 GGUF 4-bit 模型。
- 适合开发、离线演示和无 GPU 环境。

Windows 开发建议：

- 应用服务放在 Docker/WSL2。
- NVIDIA GPU 推理优先在 WSL2 或 Linux 环境。
- 不把生产部署建立在原生 Windows Worker 上。

### 11.3 模型路由

```text
规则可确定？
 ├─ 是 → 代码直接处理
 └─ 否
    ├─ 分类/抽取/改写查询 → 1.7B
    └─ 规划/综合/写作 → 4B
```

### 11.4 Model Registry 与 Model Profile

Model Registry 将 Agent 系统和后续训练完全解耦。

#### Profile

Skill、主 Agent 和子 Agent只声明逻辑能力：

```yaml
profile_id: paper_reader_v1
base_family: qwen3
size: 4b
task_type: evidence_reading
runtime:
  temperature: 0.1
  max_output_tokens: 2048
  response_schema: paper_card.schema.json
fallback_profile: general_4b_base
```

Registry 再把 Profile 解析到实际模型：

```yaml
paper_reader_v1:
  base_model: models/base/qwen3-4b
  sft_adapter: models/adapters/4b/paper_reader/sft/v1
  rl_adapter: models/adapters/4b/paper_reader/dpo/v1
  status: production
```

训练尚未完成时：

```yaml
paper_reader_v1:
  base_model: models/base/qwen3-4b
  sft_adapter: null
  rl_adapter: null
```

因此系统开发、集成测试和上线演示不依赖 SFT/RL。训练完成后，只更新 Registry 配置并通过评测晋级，不修改 Skill、子 Agent 或 Agent Runtime 代码。

当前实现使用 `models/registry.yaml` 保存 development/evaluation/production 和
`paper_reader_v1` 的逻辑映射。Manifest 只记录服务模型名、Base/SFT/RL 逻辑版本和
推理参数，不记录权重物理路径；模型调用通过 OpenAI-compatible
`/chat/completions` 协议，失败时仅使用 Profile 显式声明的 fallback。

#### Profile 绑定

| 调用方 | 默认 Profile | 基座 |
|---|---|---:|
| 意图路由器 | router_v1 | 1.7B |
| 需求澄清器 | requirement_clarifier_v1 | 4B |
| Skill 选择器 | skill_selector_v1 | 1.7B |
| 查询改写器 | query_rewriter_v1 | 1.7B |
| Tool 调用器 | tool_caller_v1 | 1.7B |
| 论文阅读子 Agent | paper_reader_v1 | 4B |
| Planner/Replanner | planner_v1 | 4B |
| 论文问答 | paper_qa_v1 | 4B |
| 多论文综合 | paper_synthesis_v1 | 4B |
| 章节/段落初次撰写 | academic_drafting_v1 | 4B |
| 章节/段落改写与润色 | academic_writer_v1 | 4B |
| 模型核验器 | verifier_v1 | 4B |

Verifier 同时保留规则/NLI 路径，不能只依赖生成模型。

### 11.6 Adapter 加载

优先使用“一个基座服务 + 多 LoRA Adapter”：

- 1.7B Adapter 只加载到 1.7B 基座。
- 4B Adapter 只加载到 4B 基座。
- 每次请求携带 `profile_id`。
- Model Router 解析 Profile 并选择对应 Adapter。
- Adapter 不兼容或加载失败时回退到对应基础 Profile。
- Trace 必须记录 base revision、SFT Adapter、RL Adapter 和 Profile 版本。

### 11.4 生成策略

- 工具调用和抽取：低温度、结构化输出。
- 事实问答：低温度，仅使用检索证据。
- 章节/段落撰写：中低温度，只使用 Writing Brief 和 Evidence Map 中的事实。
- 改写与润色：中等温度，但保护数字、公式、术语、实体和引用。
- 复杂规划：允许一次自检，不允许无限自我反思。

### 12.2 系统开发与训练解耦

两条流水线独立运行：

```text

```

约束：

- Agent Runtime 只认识 `profile_id` 和统一推理协议。
- 训练代码不导入 Agent Runtime 的业务实现，只读取导出的 Schema、Tool 定义和数据集。
- 系统开发使用基础模型、FakeLLM 或 Mock Profile。
- 新模型只有通过 Profile 对应评测后才能进入 `production.yaml`。
- 任一训练模型均可一键回滚到上一 Profile 或基础模型。

### 12.3 SFT/RL 总体顺序

阶段 J 实现状态（2026-06-21）：J01 已完成。`training/` 是独立离线包，只读取带
SHA-256 Manifest 的 Agent Schema、Tool 定义和 TrainingSample JSONL；不导入 API、
Worker、数据库、Memory、Workspace 或 Agent Runtime。数据校验强制唯一学习目标、
许可证/隐私标签、私有数据 consent_id + 匿名化，以及论文和会话级 split 隔离。
J02 已固化四个 1.7B 任务的独立数据规模、QLoRA/DPO/GRPO 方法和指标门槛，并增加
数据规模、基座 `config.json`、训练模块及显存的 fail-closed 预检。当前只有 8 GiB GPU
满足最低硬件检查，其余真实训练资产缺失，因此不产生 Adapter。

1. 使用基座模型完成系统 MVP。
2. 固定系统级和模型级评测集。
3. 训练 Reranker。
4. 对各功能 Profile 分别进行 QLoRA SFT。
5. SFT 通过后再进行 RL/偏好优化。
6. 离线评测、影子运行和 A/B 测试后晋级。

SFT 的目的，是让模型稳定学会任务格式、工具协议和领域行为。RL 的目的，是在多个“格式正确”的答案中偏向更可靠、简洁、忠于证据的答案。

### 12.4 1.7B 模型训练方案

基座：`Qwen3-1.7B` 或最终选定的同规模基座。

#### A. router/skill_selector

SFT 数据：

- 用户请求、可用 Skill 摘要、正确意图、复杂度和 Skill ID。
- 困难负例：语义相近 Skill、信息不足、无需 Agent 的请求。
- 起步 3K～5K，目标 10K～30K。

SFT 目的：

- 稳定输出意图和 Top-K Skill。
- 识别直接回答、固定工作流和动态 Agent。
- 减少把简单任务错误升级为复杂工作流。

RL：优先 DPO。

- Chosen：正确 Skill、合理复杂度和最少必要步骤。
- Rejected：错选 Skill、过度规划、漏掉必要 Skill。
- 目标：提高困难边界样本的选择质量。

评测：

- Intent Macro-F1。
- Skill Top-1、Top-3 Recall。
- 过度路由率和漏路由率。
- JSON 合法率。
- 延迟和 Token。

晋级标准：

- Top-1 ≥ 90%，Top-3 Recall ≥ 98%。
- 相比基座困难集提升 ≥ 5 个百分点。
- JSON 合法率 ≥ 99%。

#### B. query_rewriter

SFT 数据：

- 原问题、会话上下文、术语、目标检索查询。
- 包括缩写扩展、中英转换、指代消解和多跳拆分。
- 起步 2K～5K，目标 10K～20K。

SFT 目的：

- 生成更适合混合检索的查询。
- 保持原问题约束，避免引入不存在的实体。

RL：可选 DPO。

- Chosen/Rejected 由下游 Recall、MRR 和查询忠实度构造。

评测：

- 改写后的 Recall@10、MRR@10。
- 实体/约束保留率。
- 查询漂移率。

晋级标准：

- Recall@10 相对基座提升 ≥ 5%。
- 查询漂移率不高于基座。

#### C. tool_caller

SFT 数据：

- 当前状态、Skill、允许 Tool、正确 Tool 名称和参数 JSON。
- 包含不应调用 Tool、参数缺失、Tool 失败恢复样本。
- 起步 5K，目标 20K～50K。

SFT 目的：

- 在受限 Tool 集中稳定选择工具。
- 生成严格合法参数。
- 在工具失败后选择正确恢复动作。

RL：结构化任务优先 GRPO，也可先用 DPO。

可自动计算的 Reward：

- Tool 名称正确。
- 参数通过 Schema。
- 无越权字段。
- 执行结果满足预期。
- 步骤更少。

评测：

- Tool 选择准确率。
- 参数 Exact Match/字段 F1。
- Schema 合法率。
- 端到端 Tool 任务成功率。
- 多余 Tool 调用率。

晋级标准：

- Schema 合法率 ≥ 99%。
- Tool 选择准确率 ≥ 95%。
- 端到端成功率较 SFT 前提升 ≥ 5%。

### 12.5 4B 模型训练方案

基座：`Qwen3-4B` 或最终选定的同规模基座。

#### A. requirement_clarifier

SFT 数据：

- 原始需求、可用文件/Memory 摘要、信息完整性标签、缺失字段和目标问题列表。
- 正例包含真正需要澄清的论文问答、比较、写作和改写任务。
- 负例包含信息已经充分、可由 Tool 自动发现以及不应重复追问的任务。
- 起步 3K～5K，目标 10K～20K。

SFT 目的：

- 判断是否需要用户补充需求。
- 一次提出少量、相关且会影响任务结果的问题。
- 将用户回答整理成结构化 Requirement Brief。

RL：优先 DPO。

- Chosen：问题数量少、信息增益高、不询问可自动发现内容。
- Rejected：过度追问、遗漏关键问题、问题重复或与任务无关。

评测：

- Need-Clarification 分类 Macro-F1。
- 关键缺失字段 Recall。
- 不必要追问率。
- 平均问题数。
- 用户回答后任务一次进入规划的比例。
- 人工评测的问题相关性和信息增益。

晋级标准：

- Need-Clarification Macro-F1 ≥ 90%。
- 关键缺失字段 Recall ≥ 95%。
- 不必要追问率 ≤ 10%。
- 平均每轮问题数 ≤ 4。

#### B. planner/replanner

SFT 数据：

- 用户目标、当前状态、Skill、可用 Tool、结构化计划。
- 包含步骤依赖、完成条件、失败处理和重规划轨迹。
- 起步 3K～5K，目标 15K～30K。

SFT 目的：

- 生成短而可执行的计划。
- 正确表达步骤依赖、执行者和完成条件。
- 根据 Observation 进行有限重规划。

RL：优先 DPO，后续可对可执行沙箱任务使用 GRPO。

偏好/Reward：

- 计划可执行。
- 不调用不存在的 Tool。
- 依赖无环且正确。
- 成功前提下步骤更少。
- 不超过预算。

评测：

- 计划可执行率。
- 步骤依赖正确率。
- 任务完成率。
- 平均步骤、重规划率和超预算率。

晋级标准：

- 可执行率 ≥ 90%。
- 依赖正确率 ≥ 95%。
- 端到端成功率较基座提升 ≥ 5%。

#### C. paper_reader/paper_qa

SFT 数据：

- 问题、检索证据、结构化 Paper Card 或 Claim-Citation 答案。
- 包含不可回答、证据冲突、表格和跨章节样本。
- 起步 5K～10K，目标 30K～80K。

SFT 目的：

- 严格基于证据回答。
- 正确提取论文结构化字段。
- 在证据不足时拒答。

RL：优先 DPO；可自动核验部分使用 GRPO。

偏好/Reward：

- Claim 被证据蕴含。
- 引用 ID 正确。
- 数字和方向性结论正确。
- 不可回答时正确拒答。
- 无无关冗长内容。

评测：

- 问答正确率。
- Paper Card 字段 F1。
- 引用支持率。
- 数字准确率。
- 不可回答拒答率。
- 幻觉率。

晋级标准：

- 问答正确率 ≥ 80%。
- 引用支持率 ≥ 90%。
- 严重引用幻觉率 < 3%。
- 相比基座综合分提升 ≥ 5%。

#### D. paper_synthesis/academic_drafting/academic_writer

SFT 数据：

- 多篇 Paper Card、证据矩阵、目标结构和参考输出。
- 章节初稿数据包含用户提纲、Writing Brief、Evidence Map、章节类型、目标篇幅和参考初稿。
- 段落初稿数据包含论点、支撑证据、上下文衔接和目标文本。
- 学术改写数据包含原章节/段落、改写目标、不可变项和目标文本。
- 起步 8K～15K，目标 30K～80K。

SFT 目的：

- 完成多论文综合、综述草稿、章节/段落初次撰写、改写和学术润色。
- 根据章节类型遵循不同的篇章功能。
- 材料不足时输出缺失信息，而不是补造研究事实。
- 明确区分原论文结论与系统推断。
- 保持数字、公式、实体和引用。

RL：优先 DPO。

Chosen/Rejected 维度：

- 事实与证据一致。
- 覆盖用户要求。
- 章节功能与逻辑结构正确。
- 只使用用户材料和 Evidence Map 中允许的事实。
- 结构清晰。
- 语言自然但不过度改写。
- 不可变项保持。

评测：

- 综合结论引用支持率。
- 比较维度覆盖率。
- 用户材料要点覆盖率。
- 无依据事实率。
- 章节结构完整度和段落连贯性。
- 数字/公式/引用保留率。
- 语义保持和学术风格人工盲评。
- Pairwise Win Rate。

晋级标准：

- 引用支持率 ≥ 90%。
- 用户要求要点覆盖率 ≥ 90%。
- 严重无依据事实率 < 3%。
- 不可变项保留率 ≥ 99%。
- 相比 SFT 模型的 DPO 盲评胜率 ≥ 55%。

#### E. verifier

SFT 数据：

- Claim、Evidence、任务要求、支持/部分支持/不支持标签及错误类型。
- 起步 5K，目标 20K～50K。

SFT 目的：

- 识别未被证据支持的结论。
- 检查任务是否完成以及是否需要修复。

RL：仅在高质量偏好数据充足后使用 DPO；不作为第一批 RL 任务。

评测：

- 支持关系 Macro-F1。
- 严重错误 Recall。
- 误报率。
- 与人工评审一致性。

Verifier 上线必须与规则、数字校验和 NLI 组合，不能单独决定事实正确性。

### 12.6 RL 方法选择

本项目中的“RL”分两类：

1. **DPO**：主要方案。使用 Chosen/Rejected 离线偏好对，不需要在线 Reward Model 服务，适合个人项目和 1.7B/4B。
2. **GRPO**：只用于可以自动判分的任务，例如 JSON、Tool 参数、计划执行和引用 ID；Reward 必须可复现。

暂不采用 PPO：

- 训练与基础设施成本高。
- Reward Model 和在线采样复杂。
- 对本项目的首轮收益不明确。

## 16. 关键技术决策

### 决策 1：自定义状态机而不是首版直接使用 LangGraph

结论：首版自定义；后续可替换。

原因：

- 便于学习 Agent Runtime。
- 更容易限制小模型。
- 状态、重试和权限完全可见。

### 决策 2：PostgreSQL + pgvector 而不是独立向量数据库

结论：MVP 使用 pgvector。

原因：

- 减少组件。
- 权限和元数据过滤更可靠。
- 便于事务一致性。

### 决策 3：SSE 而不是 WebSocket

结论：MVP 使用 SSE。

原因：

- 服务端单向事件流已满足需求。
- 重连和基础设施更简单。

### 决策 4：4B 是主 Agent，1.7B 是辅助模型

结论：不能让 1.7B 独立承担复杂多步任务。

原因：

- 1.7B 更适合分类和抽取。
- 4B 更适合规划、综合和写作。
- 代码和核验器负责可靠性。

### 决策 5：子 Agent 是受限后台子任务

结论：不实现自由聊天式多 Agent。

原因：

- 容易评测。
- 容易并行。
- 能限制权限和预算。
